# Ignixa Upstream Gap Tracker

Running tab of Ignixa SDK gaps that block full parity in `FhirSdkMode.Ignixa`/`Hybrid`,
tracked as issues in [brendankowitz/ignixa-fhir](https://github.com/brendankowitz/ignixa-fhir).

This is narrower than [ADR-2607](adr-2607-ignixa-merge-readiness.md)'s gap register: ADR-2607
covers fhir-server-side migration work generally; this tracks only the subset that are genuine
upstream SDK defects — things Ignixa itself should fix, not things fhir-server needs to build.
Update this table whenever a new upstream gap is found or an existing issue is resolved. The `Tripwire` column records the test or signal that will mechanically indicate resolution on the next SDK bump, so gaps don't rely on memory.

| 2 | Not yet filed upstream | The `Reference` complex type's own `reference` field (a plain `string`) is itself schema-typed `IElement.InstanceType == "Reference"` instead of `"string"`. Root cause confirmed by reading `SchemaAwareElement.Children()` (`Ignixa.Serialization`, pinned SDK 0.6.7): its case-insensitive "recursive BackboneElement" heuristic -- meant for genuine self-nesting types like `QuestionnaireResponse.item.item` -- compares a child's field name against the parent type's name using `StringComparison.OrdinalIgnoreCase`. For a `Reference`-typed parent, the child field literally named `reference` matches the type name `"Reference"` case-insensitively, so the child gets stamped with the parent's own type. `GetTypeDefinition` is never called with the bare name `"reference"` in this path -- `Children()`'s qualified lookup `GetTypeDefinition("Reference.reference")` correctly returns null first. There is a SEPARATE, independent latent defect: `ISchema.GetTypeDefinition`'s backing `_types` dictionary (the generated schema providers, e.g. `R4CoreSchemaProvider`) is keyed `StringComparer.OrdinalIgnoreCase`, so a *direct* bare-name call `GetTypeDefinition("reference")` also incorrectly returns the `Reference` type -- confirmed real, but NOT the cause of this scanner's phantom element, since that call is never made on this path. Both undermine the exact `InstanceType`-based disambiguation this SDK is otherwise relied on for (see US-16 Task 1). | `IgnixaReferenceScanner` (US-16 Task 1, `src/Microsoft.Health.Fhir.Ignixa/IgnixaReferenceScanner.cs`) works around the causal (recursion-heuristic) defect with an additional `!element.HasPrimitiveValue` guard -- a real `Reference` is always JSON-object-valued; the common phantom shape (mistyped `reference` leaf holding a plain string) is JSON-string-valued, so this reliably excludes it without reintroducing name-based logic. Residual case: extension-only Reference input (`reference` string absent, but a primitive extension present via the `_reference` shadow property -- valid FHIR) produces a phantom child that is object-shaped too (no primitive value), so it passes the guard and is still yielded, with `Reference == null`. Currently harmless -- neither resolver calls `SetReference` unconditionally without checking `Reference != null` first -- but should be revisited if a future consumer calls `SetReference` unconditionally. | 0.6.7 | `IgnixaReferenceScannerTests.GivenReferenceWithPrimitiveExtensionOnItsOwnReferenceString_WhenScanned_ThenNotMistakenForNestedReference` directly probes the phantom child's `InstanceType` (asserts it still reads `"Reference"`); that assertion will FAIL once the upstream recursion heuristic is fixed -- the correct mechanical signal. (The test's original `Reference == "Practitioner/prac1"`-style assertion alone does NOT detect the phantom, since its `.Reference` reads `null` whether or not the phantom exists; mutation testing confirmed `GivenPlanDefinitionAdversarialCorpus_WhenScanned_ThenOnlyTrueReferencesAreYielded`'s `Assert.Equal(3, handles.Count)` is the assertion that actually depends on the guard's presence.) | File upstream against `SchemaAwareElement.Children()`'s case-insensitive recursive-BackboneElement name check (the causal defect) -- not `ISchema.GetTypeDefinition`'s bare-name registry lookup, which is real but non-causal on this path and can be filed/tracked separately. Once the recursion heuristic is fixed, `IgnixaReferenceScanner`'s `!HasPrimitiveValue` guard can be removed (kept until then as defense-in-depth); the guard's shadow-only residual case (see Gap column) should be re-audited at that time too. | Open |
| 3 | [ignixa-fhir#340](https://github.com/brendankowitz/ignixa-fhir/issues/340) | `BundleComponentRequestJsonNode` (`Ignixa.Serialization.Models`) exposes only `Method`/`Url` as typed properties. FHIR's `Bundle.entry.request` also defines `ifMatch`/`ifNoneMatch`/`ifModifiedSince`/`ifNoneExist` (conditional create/update/delete semantics), all missing from the typed surface -- Firely's `Bundle.RequestComponent` has all four. | `IgnixaBundleEntryView` (US-16 Task 3, `src/Microsoft.Health.Fhir.Shared.Api/Features/Resources/Bundle/IgnixaBundleEntryView.cs`) reads these four properties directly off `BundleComponentRequestJsonNode.MutableNode` (confirmed public on the pinned 0.6.7 package, matching the established raw-node-property-access pattern already used by `IgnixaImportResourceParser`) instead of a typed accessor. | 0.6.7 | Once `ignixa-fhir#340` closes and typed `IfMatch`/`IfNoneMatch`/`IfModifiedSince`/`IfNoneExist` properties land on `BundleComponentRequestJsonNode`, `IgnixaBundleEntryView`'s private `GetRequestHeaderProperty` raw-node reads can be replaced with direct typed-property reads -- no test currently fails on this (both paths produce identical values), so this is a manual follow-up, not an automatic tripwire. | Replace `IgnixaBundleEntryView`'s four `GetRequestHeaderProperty(...)` calls with the new typed properties and delete the helper. Also worth checking at that time whether the SDK has by then made `MutableNode` `internal` (already true on the `ignixa-fhir` main branch, ahead of 0.6.7) -- if so, any other remaining raw-node access in fhir-server needs to switch to the `IMutableJsonNode` explicit-interface-cast pattern. | Open |
