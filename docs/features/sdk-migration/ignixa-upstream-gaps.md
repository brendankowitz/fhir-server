# Ignixa Upstream Gap Tracker

Running tab of Ignixa SDK gaps that block full parity in `FhirSdkMode.Ignixa`/`Hybrid`,
tracked as issues in [brendankowitz/ignixa-fhir](https://github.com/brendankowitz/ignixa-fhir).

This is narrower than [ADR-2607](adr-2607-ignixa-merge-readiness.md)'s gap register: ADR-2607
covers fhir-server-side migration work generally; this tracks only the subset that are genuine
upstream SDK defects — things Ignixa itself should fix, not things fhir-server needs to build.
Update this table whenever a new upstream gap is found or an existing issue is resolved. The `Tripwire` column records the test or signal that will mechanically indicate resolution on the next SDK bump, so gaps don't rely on memory.

| 2 | Not yet filed upstream | The `Reference` complex type's own `reference` field (a plain `string`) is itself schema-typed `IElement.InstanceType == "Reference"` instead of `"string"`. Confirmed empirically: `ISchema.GetTypeDefinition("reference")` (lowercase, the field name) returns the `Reference` *type* definition -- an apparent case-insensitive collision in the schema's type registry between the field name "reference" and the datatype name "Reference" (`GetTypeDefinition("Reference.reference")`, the BackboneElement-qualified lookup, correctly returns null; the bad match happens on the bare-name fallback). This undermines the exact `InstanceType`-based disambiguation this SDK is otherwise relied on for (see US-16 Task 1). | `IgnixaReferenceScanner` (US-16 Task 1, `src/Microsoft.Health.Fhir.Ignixa/IgnixaReferenceScanner.cs`) works around it with an additional `!element.HasPrimitiveValue` guard -- a real `Reference` is always JSON-object-valued, the mistyped `reference` leaf is always JSON-string-valued, so this reliably excludes the false match without reintroducing name-based logic. Functionally harmless for reference resolution itself (the phantom element's `Meta<JsonNode>()` backing is either absent or lacks a "reference" key, so it's always a no-op even without the guard) -- but any other Ignixa consumer walking the schema-typed tree by `InstanceType == "Reference"` without this guard would hit the same false positive. | 0.6.7 | `IgnixaReferenceScannerTests.GivenReferenceWithPrimitiveExtensionOnItsOwnReferenceString_WhenScanned_ThenNotMistakenForNestedReference` pins this down; if it starts failing because the phantom disappears, the guard has become redundant (harmless to leave, but confirms this gap has closed). | File upstream against `SchemaAwareElement`/`ISchema.GetTypeDefinition`'s bare-name fallback; once fixed, `IgnixaReferenceScanner`'s `!HasPrimitiveValue` guard can be removed (kept until then as defense-in-depth). | Open |
